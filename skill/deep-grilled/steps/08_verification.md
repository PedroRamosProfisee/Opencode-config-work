# Step 08 — Verification

## Goal

Define how we prove the idea, product, feature, or implementation works.

## Two verification tracks

### Product/business validation

For market and monetization:

```markdown
Experiment:
Hypothesis:
Audience:
Method:
Timebox:
Success metric:
Failure metric:
Cost:
Decision unlocked:
```

### Engineering validation

For implementation:

```markdown
Acceptance criterion:
Given:
When:
Then:
Test type:
QA scenario:
Failure route: requirement gap / design mismatch / implementation bug / environment issue
```

## CodeMySpec-style artifact mapping

Use these if the plan may move into build:

- User stories
- Acceptance criteria
- ADR candidates
- Component map
- Spec stubs
- BDD scenarios
- Test plan
- QA checklist
- Issue routing rules

## Recommended default

Before building an indie/SaaS MVP, require at least one non-code validation:

- 5 customer conversations, or
- 10 qualified waitlist leads, or
- 2 paid pilot commitments, or
- a landing page with meaningful conversion from a realistic channel.
